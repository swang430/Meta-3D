## 1. 产品概述

### 1.1 产品定位

**虚拟路测平台（Virtual Road Test Platform）** 是Meta-3D MIMO OTA测试系统的核心扩展功能，旨在通过三种不同精度和成本的测试模式，在实验室环境中复现真实道路测试场景，支持车联网设备的研发验证和认证测试。

### 1.2 核心价值主张

| 维度 | 传统路测 | 虚拟路测平台 |
|------|---------|-------------|
| **成本** | 高（车辆、人力、场地） | 低（实验室设备） |
| **周期** | 长（天/周） | 短（分钟/小时） |
| **场景覆盖** | 有限（地理限制） | 全面（无限场景） |
| **可重复性** | 差（环境随机） | 优（确定性环境） |
| **测试精度** | 真实场景 | 渐进验证（仿真→传导→OTA） |
| **24/7测试** | 困难 | 支持 |
| **极端场景** | 危险/不可行 | 安全可控 |

### 1.3 应用场景

- **研发阶段**: 快速迭代验证设计方案
- **功能测试**: 切换、波束管理、载波聚合等
- **性能测试**: 吞吐量、延迟、覆盖率
- **认证测试**: 3GPP、CTIA、5GAA标准符合性
- **回归测试**: 软件更新后的自动化验证
- **极端场景**: 隧道、高速、密集城区等

## 2. 核心概念

### 2.1 测试模式 (Test Mode)

虚拟路测平台支持三种测试模式，形成**渐进验证路径**：

```
成本低 ←──────────────────────────────→ 成本高
精度中 ←──────────────────────────────→ 精度高

[全数字仿真] → [传导测试] → [OTA辐射测试] → [真实路测]
 Digital Twin   Conducted      MPAC OTA       Real Road
```

### 2.2 路测场景 (Road Test Scenario)

路测场景是测试的核心输入，包含：

- **网络配置**: 5G NR/LTE/C-V2X，频段，带宽，基站位置
- **路径轨迹**: 车辆移动路径，速度曲线，时间戳
- **环境条件**: 城市/高速/隧道，天气，交通密度，障碍物
- **业务流量**: FTP/视频/VoIP，上下行，数据速率
- **触发事件**: 切换、波束切换、干扰等
- **KPI定义**: 吞吐量、延迟、BLER目标值

**一次定义，三处使用** - 同一场景可在三种测试模式下执行。

### 2.3 网络拓扑 (Network Topology)

描述测试设备的连接关系（传导模式专用）：

- **基站仿真器**: CMX500, 8820C
- **信道仿真器**: PropsIM F64, Vertex, KSW-WNS02B
- **DUT**: 车辆，天线端口数
- **RF连接**: 端口映射，线缆规格，损耗补偿

## 3. 三种测试模式

### 3.1 模式对比

| 对比维度 | 全数字仿真 | 传导测试 | OTA辐射测试 |
|---------|-----------|---------|------------|
| **硬件需求** | 无（纯软件） | 仪表+DUT | 仪表+DUT+MPAC暗室 |
| **成本** | 💰 低 | 💰💰 中 | 💰💰💰 高 |
| **测试周期** | ⚡ 分钟级 | ⏱️ 小时级 | 🕐 天级 |
| **场景灵活性** | ⭐⭐⭐ 无限 | ⭐⭐ 有限 | ⭐⭐ 有限 |
| **测试精度** | ⭐⭐ 中 | ⭐⭐⭐ 高 | ⭐⭐⭐⭐ 最高 |
| **实时性** | 可加速10x | 实时 | 实时 |
| **并行能力** | 多场景并行 | 有限 | 单场景 |
| **认证级别** | 不适用 | 部分适用 | 完全适用 |

### 3.2 模式1: 全数字仿真 (Digital Twin)

**原理**: 网络设备、信道环境、被测物体全部数字化，在算力平台完成性能仿真。

**架构**:
```
┌─────────────┐   ┌─────────────┐   ┌─────────────┐
│ 网络仿真器   │   │ 信道仿真器   │   │  DUT仿真器   │
│ gNB/eNB模型 │←→│ 信道模型     │←→│ 车辆平台模型  │
│ 协议栈      │   │ 多径传播     │   │ 天线阵列     │
└─────────────┘   └─────────────┘   └─────────────┘
                         ↓
                  ┌─────────────┐
                  │  场景引擎    │
                  │ 轨迹生成     │
                  │ 时间推进     │
                  └─────────────┘
                         ↓
                  [算力平台: GPU/CPU]
```

**适用场景**:
- ✅ 早期研发阶段，快速验证想法
- ✅ 算法优化，需要大量测试迭代
- ✅ 极端场景探索（隧道、高速、干扰）
- ✅ 参数扫描（频率、功率、天线配置）
- ❌ 最终认证测试

**关键参数**:
- 仿真精度: `fast` (10x加速) / `balanced` (实时) / `accurate` (0.1x慢速)
- 信道抽头数: 16/64/256
- 时间步长: 10ms / 1ms / 0.1ms

### 3.3 模式2: 传导测试 (Conducted)

**原理**: 真实仪表/网络设备与DUT通过射频线缆直连，在屏蔽暗室中测试。

**架构**:
```
[基站仿真器] ──RF线缆──→ [信道仿真器] ──RF线缆──→ [DUT天线端口]
   CMX500              PropsIM F64            车辆TCU
   4 TX端口            4输入 → 64输出         4 RX端口

                    [屏蔽暗室环境]
```

**适用场景**:
- ✅ 功能验证（切换、波束管理）
- ✅ 性能调试（吞吐量、延迟优化）
- ✅ 问题定位（隔离空间干扰）
- ✅ 中间验证阶段
- ❌ 天线辐射特性测试

**关键特性**:
- 拓扑配置器: 可视化配置RF连接
- 线缆损耗补偿: 自动校准
- DUT控制: ADB/USB/Ethernet多种接口
- 实时监控: 10Hz采样率

### 3.4 模式3: OTA辐射测试 (MPAC OTA)

**原理**: 复用现有MPAC架构，增加路测场景支持，通过32探头暗室进行空中辐射测试。

**架构**:
```
[基站仿真器] → [信道仿真器] → [功放阵列] → [32探头] ⟿⟿⟿ [DUT车辆]
                                                  空间辐射

                              [MPAC暗室 + 转台]
```

**适用场景**:
- ✅ 最终认证测试（3GPP, CTIA）
- ✅ 天线性能测试（Pattern, 增益）
- ✅ 整车OTA性能
- ✅ 与真实路测对标

**场景映射**:
- 轨迹 → 转台运动（方位角、俯仰角）
- 环境 → 信道模型（UMa, UMi, RMa）
- 速度 → 多普勒效应
- 基站位置 → 波束方向

## 4. 系统架构

### 4.1 整体架构

```
┌─────────────────────────────────────────────────────────────┐
│                     前端层 (React GUI)                        │
│  模式选择 | 场景库 | 场景编辑器 | 拓扑配置 | 实时监控 | 结果分析 │
└──────────────────────────┬──────────────────────────────────┘
                           │ REST API / WebSocket
┌──────────────────────────┴──────────────────────────────────┐
│                   服务编排层 (Backend)                        │
│  测试计划管理 | 执行引擎 | 模式切换 | 资源调度 | 数据采集      │
└──────────────────────────┬──────────────────────────────────┘
                           │
        ┌──────────────────┼──────────────────┐
        ▼                  ▼                  ▼
┌───────────────┐  ┌───────────────┐  ┌───────────────┐
│数字孪生执行器  │  │传导测试执行器  │  │ OTA执行器     │
│DigitalTwin    │  │ Conducted     │  │ OTA           │
│Executor       │  │ Executor      │  │ Executor      │
└───────────────┘  └───────────────┘  └───────────────┘
```

### 4.2 统一测试抽象层

**核心设计**: 所有测试模式实现统一的 `ITestExecutor` 接口

```typescript
interface ITestExecutor {
  // 生命周期
  initialize(config: TestConfig): Promise<TestResult>;
  execute(): Promise<ExecutionHandle>;
  pause/resume/stop(): Promise<void>;
  cleanup(): Promise<void>;

  // 配置
  loadScenario(scenario: RoadTestScenario): Promise<TestResult>;
  configureTopology(topology: NetworkTopology): Promise<TestResult>;

  // 监控
  queryStatus(): Promise<TestStatus>;
  getMetrics(): Promise<TestMetrics>;
}
```

**优势**:
- ✅ 模式无缝切换（同一测试计划可在不同模式执行）
- ✅ 前端代码复用（同一UI适配所有模式）
- ✅ 易于扩展（添加新模式只需实现接口）

## 5. 核心功能

### 5.1 场景库 (Scenario Library)

**预定义场景分类**:

1. **标准场景** (Standard)
   - 3GPP: UMa, UMi, RMa, InH
   - CTIA: OTA性能测试例
   - 5GAA: V2X场景

2. **功能场景** (Functional)
   - 小区切换 (Handover)
   - 波束切换 (Beam Switching)
   - 载波聚合 (Carrier Aggregation)

3. **性能场景** (Performance)
   - 最大吞吐量
   - 低延迟
   - 超高可靠性 (URLLC)

4. **环境场景** (Environment)
   - 城市峡谷 (Urban Canyon)
   - 高速公路 (Highway)
   - 隧道 (Tunnel)
   - 停车场 (Parking Lot)

5. **极端场景** (Extreme)
   - 小区边缘 (Cell Edge)
   - 高干扰 (High Interference)
   - 快速衰落 (Fast Fading)

6. **自定义场景** (Custom)
   - 用户通过场景编辑器创建

**场景数据结构**:
```typescript
interface RoadTestScenario {
  id: string;
  name: string;
  category: ScenarioCategory;
  source: 'standard' | 'custom';

  network: NetworkConfig;      // 网络配置
  route: Route;                // 轨迹
  environment: Environment;    // 环境
  traffic: TrafficConfig;      // 业务流量
  events: ScenarioEvent[];     // 触发事件
  kpi: KPIDefinition[];        // KPI目标
}
```

### 5.2 场景编辑器 (Scenario Editor)

可视化编辑器，支持：

- **地图编辑器**: 基于OpenStreetMap，拖拽绘制轨迹
- **网络配置器**: 放置基站，配置参数
- **环境编辑器**: 添加建筑物、树木等障碍物
- **事件时间线**: 时间轴上添加触发事件
- **KPI定义器**: 设置性能指标目标值

### 5.3 拓扑配置器 (Topology Configurator)

用于传导测试模式的RF连接配置工具：

**三种编辑模式**:
1. **向导模式**: 问答式引导配置
2. **可视化模式**: 拖拽连接设备
3. **代码模式**: 直接编辑JSON

**功能**:
- 设备库: CMX500, PropsIM F64, Vertex等
- 端口映射: 可视化连接源端口→目标端口
- 线缆管理: RG-58, LMR-400规格库
- 损耗补偿: 自动计算链路损耗
- 拓扑验证: 检查连接有效性

### 5.4 实时监控 (Real-time Monitoring)

**数字孪生仪表板**:
- 仿真时钟: 仿真时间 vs 真实时间
- 3D可视化: 车辆轨迹、基站位置、信号覆盖
- 实时KPI: 吞吐量、延迟、SINR、BLER曲线
- 控制面板: 暂停/继续、速度调节

**传导/OTA监控**:
- 仪表状态: 温度、告警
- RF功率: 实时功率监控
- DUT状态: 连接状态、日志流
- KPI仪表盘: 实时图表

### 5.5 结果分析 (Result Analysis)

**对比分析**:
- 模式间对比: 同一场景在三种模式下的KPI对比
- 场景对比: 不同场景的性能对比
- 时间序列分析: KPI随时间变化趋势

**报告生成**:
- PDF报告: 符合3GPP格式
- HTML交互报告: 可缩放图表
- 数据导出: CSV, JSON, MATLAB格式

## 6. 用户工作流

### 6.1 典型工作流

```
1. 选择测试模式
   ↓
2. 选择/创建路测场景
   ↓
3. [传导模式] 配置网络拓扑
   ↓
4. 配置DUT参数
   ↓
5. 定义KPI目标
   ↓
6. 执行测试
   ↓
7. 实时监控
   ↓
8. 结果分析
   ↓
9. 生成报告
```

### 6.2 渐进验证工作流

**研发迭代流程**:

```
阶段1: 快速探索 (1天)
  全数字仿真 × 50个场景 → 筛选出10个关键场景

阶段2: 功能验证 (3天)
  传导测试 × 10个关键场景 → 发现5个问题场景

阶段3: 问题定位 (2天)
  数字仿真 + 传导测试 × 5个问题场景 → 优化DUT算法

阶段4: 回归测试 (1天)
  传导测试 × 10个场景 → 确认问题修复

阶段5: 认证准备 (5天)
  OTA测试 × 3GPP标准场景 → 生成认证报告
```

## 7. 实施路线图

### 7.1 Phase 1: 基础架构 (4-6周)

**目标**: 建立统一测试抽象层和前端框架

**交付物**:
- ✅ `ITestExecutor` 接口定义
- ✅ 数据模型: `RoadTestScenario`, `NetworkTopology`
- ✅ API规范扩展 (OpenAPI YAML)
- ✅ 前端页面: 模式选择、场景库（Mock数据）
- ✅ Mock服务器扩展

### 7.2 Phase 2: OTA模式增强 (3-4周)

**目标**: 复用现有MPAC架构，增加路测场景支持

**交付物**:
- ✅ `OTAScenarioMapper` - 场景到OTA配置的映射
- ✅ 场景库 - 至少5个标准场景
- ✅ 场景编辑器 - 基础版
- ✅ 与现有 `ChannelEmulatorHAL` 集成

### 7.3 Phase 3: 传导测试模式 (6-8周)

**目标**: 实现仪表-DUT传导测试

**交付物**:
- ✅ `IBaseStationDriver`, `IDUTDriver` 接口
- ✅ CMX500驱动实现（或Mock）
- ✅ DUT控制实现 (ADB/USB)
- ✅ `TopologyManager` - 拓扑管理
- ✅ `ConductedExecutor` - 传导测试执行器
- ✅ 拓扑配置器UI

### 7.4 Phase 4: 数字孪生模式 (8-12周)

**目标**: 实现纯软件仿真

**交付物**:
- ✅ 场景引擎 - 轨迹生成、时间推进
- ✅ 网络仿真器 - gNB/eNB模型
- ✅ 信道仿真器 - 3GPP信道模型
- ✅ DUT仿真器 - 车辆平台模型
- ✅ `DigitalTwinExecutor` - 数字孪生执行器
- ✅ 3D可视化 (Three.js)
- ✅ GPU加速优化

### 7.5 Phase 5: 高级功能 (4-6周)

**目标**: 完善用户体验和高级功能

**交付物**:
- ✅ 场景编辑器增强 - 地图集成
- ✅ 结果分析仪表板
- ✅ 自动化报告生成
- ✅ 3GPP标准场景验证
- ✅ 用户文档和培训材料

### 7.6 时间线总结

| 阶段 | 周数 | 累计 | 关键里程碑 |
|-----|------|------|-----------|
| Phase 1 | 4-6 | 6周 | 基础架构完成 |
| Phase 2 | 3-4 | 10周 | OTA模式可用 |
| Phase 3 | 6-8 | 18周 | 传导模式可用 |
| Phase 4 | 8-12 | 30周 | 数字孪生可用 |
| Phase 5 | 4-6 | 36周 | 产品化完成 |

**总计**: 24-36周 (6-9个月)

## 8. 成功指标

### 8.1 功能指标

- [ ] 三种测试模式均可正常运行
- [ ] 模式间可无缝切换（同一场景）
- [ ] 支持 ≥20 个标准场景
- [ ] 支持自定义场景创建和编辑
- [ ] 拓扑配置器支持 SISO 到 8×8 MIMO
- [ ] 实时监控延迟 < 1秒

### 8.2 性能指标

**数字孪生模式**:
- 1x实时仿真: 4×4 MIMO, 100MHz带宽, 64信道抽头
- 10x加速仿真: 2×2 MIMO, 20MHz带宽, 16信道抽头
- GPU利用率 > 70%

**传导测试模式**:
- 仪表控制响应时间 < 500ms
- KPI采样率 ≥ 10 Hz
- 支持多DUT并发测试 ≥ 4台

**OTA测试模式**:
- 复用现有MPAC性能指标
- 场景映射准确度 > 90%

### 8.3 精度指标

- **模式对比**: 同一场景在三种模式下的KPI偏差 < 20%
- **标准符合性**: 3GPP标准场景结果符合规范要求
- **可重复性**: 同一测试重复10次，标准差 < 5%

### 8.4 业务指标

- **成本降低**: 相比真实路测，成本降低 > 60%
- **周期压缩**: 相比真实路测，周期压缩 > 80%
- **场景覆盖**: 标准场景覆盖率 > 90%
- **用户满意度**: NPS > 8/10

## 9. 风险与挑战

### 9.1 技术风险

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| 数字孪生仿真精度不足 | 高 | 分级精度模型，与真实测试对标校准 |
| 性能不满足实时要求 | 中 | GPU加速，时间压缩模式 |
| DUT控制接口多样性 | 中 | 插件化驱动架构 |
| 场景数据管理复杂 | 低 | 分层存储，懒加载 |

### 9.2 业务风险

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| 用户接受度低 | 中 | 提供渐进验证路径，降低迁移门槛 |
| 标准符合性问题 | 高 | 严格遵循3GPP规范，第三方验证 |
| 竞品压力 | 低 | 三模式一体化是差异化优势 |
